| Challenge                               | Risk                                                                                                  | How to Tackle It                                                                                                                                        |
| --------------------------------------- | ----------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **1. Learning Curve of ERPNext/Frappe** | Team spends too much time understanding internals instead of building features.                       | Stick to standard ERPNext DocTypes wherever possible. Customize only when absolutely necessary. Avoid modifying ERPNext core.                           |
| **2. Scope Creep**                      | New requirements keep getting added, making the August deadline impossible.                           | Freeze the scope. Build only the agreed modules. Push enhancements (mobile app, advanced analytics, AI, etc.) to Phase 2.                               |
| **3. React ↔ ERPNext Integration**      | Authentication, API routing, and CORS issues delay development.                                       | Build the integration layer first (login, API helper, proxy configuration) before developing any UI pages. Validate it with a simple Asset List screen. |
| **4. Dual Backend Management**          | Maintaining Django and ERPNext simultaneously increases complexity.                                   | Clearly separate responsibilities: Django handles PMS, ERPNext handles Asset modules. Avoid cross-database dependencies initially.                      |
| **5. Authentication Between Systems**   | Users may need to log in twice or sessions may become inconsistent.                                   | Implement Single Sign-On (or token sharing) later if time permits. For MVP, ensure a seamless login flow with synchronized user accounts.               |
| **6. ERPNext Customization Complexity** | Over-customization creates upgrade and maintenance issues.                                            | Prefer Workflows, Custom Fields, Property Setters, and Server Scripts. Create custom DocTypes only when ERPNext cannot support the requirement.         |
| **7. API Performance**                  | Dashboard pages requiring many API calls become slow.                                                 | Fetch only required fields, paginate lists, cache dashboard data, and avoid unnecessary joins.                                                          |
| **8. UI Consistency**                   | React UI and ERPNext concepts may feel disconnected.                                                  | Keep React as the only frontend. ERPNext acts purely as the backend. Use a common design system and reusable components.                                |
| **9. Workflow Configuration**           | Approval workflows can consume significant time.                                                      | Start with a minimal approval flow. Add complex approval matrices only where required.                                                                  |
| **10. Barcode/QR Integration**          | Scanner compatibility and browser permissions may cause issues.                                       | Implement scanning only for critical workflows first (GRN, Asset Transfer, Stock Issue). Use proven browser-based scanning libraries.                   |
| **11. Data Migration**                  | Existing PMS or master data may need synchronization with ERPNext.                                    | Import only essential master data initially (Users, Suppliers, Assets, Warehouses). Delay full synchronization until after MVP.                         |
| **12. Reporting & Analytics**           | Custom reports often take longer than expected.                                                       | Use ERPNext's built-in reports where possible. Develop only the dashboards required for demonstration.                                                  |
| **13. Testing Across Two Systems**      | Bugs appear because React, Django, and ERPNext evolve independently.                                  | Test module by module. Complete integration testing after each feature instead of waiting until the end.                                                |
| **14. Deployment Complexity**           | Running React, Django, ERPNext, Redis, MariaDB, and workers together can introduce deployment issues. | Finalize the deployment architecture early. Set up staging well before feature completion and deploy regularly.                                         |
| **15. Team Coordination**               | Backend and frontend development block each other.                                                    | Define APIs first. Frontend can develop against mock data while ERPNext APIs are being implemented.                                                     |
